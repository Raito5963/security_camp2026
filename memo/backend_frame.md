バックエンドのシステム構成図
ユーザーがWebアプリで「画像を投稿する」という動作をした時、裏側では以下のような連携が行われています。

Frontend (Next.js / Vercel): ユーザーが画像を選択し、「投稿」ボタンを押す。

API Gateway / Server (Go + Gin / Docker): リクエストを受け取り、認証（Supabase Authなど）を確認する。

Storage (MinIO): 画像本体を保存し、保存先のURLを取得する。

Database (PostgreSQL): 「誰が、いつ、どのURLの画像を投稿したか」という情報を保存する。

Cache (Redis): 人気の投稿であれば、毎回DBを見に行かずに済むようデータを一時保存して高速化する。


システム構成例：Instagram風SNS
各スタックの役割分担
Frontend (Next.js / Vercel)

ユーザーが目にする画面。画像の選択、フィルター加工、タイムラインの表示。

Vercelにデプロイし、エッジネットワークで高速に配信。

Backend API (Go + Gin / Docker)

役割: 投稿リクエストの受付、ユーザー認証の検証、DBへの保存指示。

メリット: Goの並列処理（Goroutine）を活かし、大量のユーザーからの「いいね」やコメントを効率よくさばく。

Database (PostgreSQL)

役割: ユーザー情報、フォロー関係、投稿のキャプション、保存された画像のURL管理。

ポイント: リレーショナルDB（RDB）を使うことで、「誰が誰をフォローしているか」といった複雑な関係性を正確に管理。

Object Storage (MinIO)

役割: アップロードされた「画像ファイルそのもの」の保存。

ポイント: DBには「画像のパス（場所）」だけを書き込み、実体はここに置くことでDBの肥大化を防ぎ、コストも抑制。

Cache (Redis)

役割: ログインセッションの保持や、タイムラインの「最新10件」などの一時保存。

メリット: 人気ユーザーの投稿など、アクセスが集中するデータを高速に返せる。
